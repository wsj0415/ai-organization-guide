# 第 4 章：技术栈选择

> **核心洞察:** 选对技术栈，AI 组织成功一半。选错了，后期重构成本是初期的 10 倍。

---

## 4.1 技术栈选型原则

2026 年，AI 技术栈的选择比 2025 年复杂 10 倍。

2025 年，你只需要选：GPT-4 还是 Claude。  
2026 年，你要选：
- LLM 模型（10+ 主流选项）
- Agent 框架（20+ 开源项目）
- 向量数据库（15+ 服务商）
- 部署平台（云厂商 + 边缘计算）
- 监控工具（可观测性栈）

**选择恐惧症？正常。**

本章给你一套选型框架，帮你避开 90% 的坑。

## 4.2 LLM 模型选型

### 主流模型对比（2026 年 Q1）

| 模型 | 优势 | 劣势 | 适用场景 | 成本 |
|------|------|------|----------|------|
| **GPT-4.5** | 综合最强，生态完善 | 贵，审查严格 | 通用场景，企业级 | $$$$ |
| **Claude 3.5** | 长文本强，安全友好 | 多模态弱 | 文档分析，内容创作 | $$$ |
| **Gemini 3 Pro** | 多模态最强，Google 生态 | 中文略弱 | 图像 + 文本混合 | $$$ |
| **GLM-4.7** | 中文最强，本地部署 | 英文略弱 | 中文场景，私有化 | $$ |
| **Qwen 3.5** | 性价比高，开源 | 生态较小 | 预算有限，定制化 | $ |
| **Llama 4** | 开源免费，可微调 | 需要自部署 | 数据敏感，定制化 | $（硬件成本） |

### 选型决策树

```
问自己 3 个问题：

1. 预算充足吗？
   - 是 → GPT-4.5 / Claude 3.5
   - 否 → Qwen 3.5 / Llama 4

2. 主要语言是中文吗？
   - 是 → GLM-4.7 / Qwen 3.5
   - 否 → GPT-4.5 / Claude 3.5

3. 需要本地部署吗？
   - 是 → Llama 4 / Qwen 开源版
   - 否 → 云服务（GPT/Claude/Gemini）
```

### 推荐组合（2026 年）

**方案 A：企业级（预算充足）**
```
主力：GPT-4.5（复杂任务）
备用：Claude 3.5（长文本）
监控：双模型输出对比
成本：$0.05-0.1/千 tokens
```

**方案 B：性价比（推荐）**
```
主力：Qwen 3.5 Plus（80% 任务）
备用：GPT-4.5（20% 复杂任务）
成本：$0.01-0.03/千 tokens
节省：60-70%
```

**方案 C：私有化（数据敏感）**
```
主力：Llama 4 70B（本地部署）
微调：行业数据 LoRA
成本：硬件 $10,000+ 电费
优势：数据不出域
```

> **阶段 1（2025 Q3）：盲目追新**

>

> - 复杂任务：GPT-4.5（20%）
> - 中等任务：Qwen 3.5（50%）
> - 简单任务：Llama 4（30%，本地）
> - 月成本：$2,000
> - 节省：75%
> 
> **关键洞察：** 不是越贵越好，是"合适最好"。

## 4.3 Agent 框架选型

### 主流框架对比

| 框架 | 语言 | 优势 | 劣势 | 成熟度 |
|------|------|------|------|--------|
| **LangChain** | Python/JS | 生态最大，组件多 | 重，学习曲线陡 | ⭐⭐⭐⭐⭐ |
| **AutoGen** | Python | 多 Agent 协作强 | 文档一般 | ⭐⭐⭐⭐ |
| **OpenClaw** | Node.js | 轻量，易上手 | 生态较小 | ⭐⭐⭐ |
| **CrewAI** | Python | 角色分工清晰 | 灵活性一般 | ⭐⭐⭐⭐ |
| **LlamaIndex** | Python | 数据索引强 | Agent 功能弱 | ⭐⭐⭐⭐ |

### 选型建议

**个人/小团队 → OpenClaw**
- 优点：轻量，配置简单，10 分钟上手
- 缺点：复杂场景支持有限
- 适用：客服/内容/数据等标准场景

**中大型企业 → LangChain**
- 优点：组件丰富，可定制性强
- 缺点：重，需要专业团队
- 适用：复杂业务逻辑，多系统集成

**多 Agent 协作 → AutoGen/CrewAI**
- 优点：原生支持多 Agent 对话
- 缺点：调试复杂
- 适用：需要多个 AI 角色协作的场景

### 案例：某内容工厂的框架演进

> **阶段 1：LangChain（过度设计）**
> ```python
> # 简单的内容生成，写了 200 行配置
> agent = Agent(
>     role="Content Writer",
>     goal="Write blog posts",
>     backstory="...",
>     verbose=True,
>     allow_delegation=False,
>     tools=[...],
>     llm=...,
>     memory=...,
>     callbacks=[...]
> )
> ```
> - 问题：杀鸡用牛刀，维护成本高
> 
> **阶段 2：OpenClaw（刚刚好）**
> ```javascript
> // 同样功能，10 行配置
> const agent = {
>   role: "content-writer",
>   task: "generate-blog-post",
>   model: "qwen-3.5-plus"
> };
> ```
> - 效果：功能一样，代码少 95%

## 4.4 数据存储选型

### 需要哪些存储？

AI 组织需要 3 种存储：

**1. 向量数据库（记忆）**
- 用途：存储历史对话、知识库
- 场景：RAG（检索增强生成）
- 选项：
  - Pinecone（云托管，贵）
  - Weaviate（开源，可自部署）
  - Chroma（轻量，适合开发）
  - Qdrant（性能好，推荐）

**2. 关系数据库（结构化数据）**
- 用途：用户数据、订单、配置
- 场景：传统业务数据
- 选项：
  - PostgreSQL（推荐）
  - MySQL（成熟）
  - Supabase（PostgreSQL + 实时）

**3. 对象存储（文件）**
- 用途：图片、文档、音频
- 场景：多媒体内容
- 选项：
  - AWS S3（标准）
  - Cloudflare R2（便宜，无出口费）
  - 阿里云 OSS（国内）

### 推荐组合

**初创团队（最小成本）**
```
向量：Chroma（本地，免费）
关系：SQLite（本地，免费）
对象：Cloudflare R2（10GB 免费）
月成本：$0
```

**成长团队（性价比）**
```
向量：Qdrant Cloud（$20/月）
关系：Supabase（$25/月）
对象：Cloudflare R2（$5/月）
月成本：$50
```

**企业团队（高可用）**
```
向量：Pinecone（$200/月）
关系：AWS RDS PostgreSQL（$100/月）
对象：AWS S3（$50/月）
月成本：$350
```

## 4.6 监控与可观测性

### 需要监控什么？

**1. 模型性能**
- 响应时间
- Token 使用量
- 错误率
- 成本追踪

**2. Agent 行为**
- 任务完成率
- 自主决策次数
- 升级人类比例
- 学习优化效果

**3. 业务指标**
- 客户满意度
- 转化率
- 人力节省比例
- ROI

### 监控工具

**开源方案**
```
日志：Winston + File
指标：Prometheus + Grafana
追踪：Jaeger
成本：免费（需要自部署）
```

**云方案**
```
日志：Logtail（$20/月）
指标：DataDog（$50/月）
追踪：Sentry（$25/月）
成本：$95/月
```

**简化方案（推荐）**
```
日志：文件日志 + 每日汇总
指标：Notion 数据库手动记录
成本：$0
时间：10 分钟/天
```

## 4.8 选型避坑指南

### 坑 1：过度设计

**错误：** 第一天就用 LangChain + Pinecone + AWS  
**正确：** OpenClaw + Chroma + SQLite，验证后再升级

**教训：** 技术栈应该跟随业务增长，不是领先业务。

---

### 坑 2：锁定单一供应商

**错误：** 全部用 GPT-4，某天 API 挂了全瘫痪  
**正确：** GPT-4（主力）+ Qwen（备用），自动切换

**教训：** 永远要有 Plan B，最好 Plan C。

---

### 坑 3：忽视成本监控

**错误：** 上线 1 个月，收到 $10,000 账单  
**正确：** 每日成本告警，超过$100/天自动降级

**教训：** 设置预算告警，比事后哭诉有用。

---

### 坑 4：盲目追求最新

**错误：** 每个新模型都试，生产环境不稳定  
**正确：** 稳定模型为主（80%），新模型灰度测试（20%）

**教训：** 生产环境要稳定，不是最新。

## 4.9 本章小结

### 核心要点

✅ **LLM 选型：合适最好，不是最贵最好**
- 中文场景：GLM-4.7 / Qwen 3.5
- 英文场景：GPT-4.5 / Claude 3.5
- 预算有限：Qwen 3.5 / Llama 4

✅ **Agent 框架：简单优先**
- 个人/小团队：OpenClaw
- 中大型：LangChain
- 多 Agent：AutoGen/CrewAI

✅ **数据存储：按需选择**
- 向量：Chroma（免费）→ Qdrant（生产）
- 关系：SQLite → PostgreSQL
- 对象：Cloudflare R2

✅ **部署平台：快速上线**
- 国际：Vercel + Railway
- 国内：阿里云
- 大规模：AWS

✅ **监控：从简到繁**
- 初期：文件日志 + 手动记录
- 生产：Sentry + DataDog

### 行动清单

1. 用决策树选择 LLM 模型
2. 根据团队规模选择 Agent 框架
3. 选择数据存储方案（最小可行 → 生产）
4. 配置部署平台
5. 设置成本告警（重要！）
